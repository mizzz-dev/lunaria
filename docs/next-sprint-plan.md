# 現状サマリー

- Lunaria は Discord コミュニティ運営向け Bot + 管理ダッシュボードとして、MVP の主要機能が一通り実装済み。基本ユースケース（自動応答、モデレーション、イベント運営、分析の取得）は成立している。  
- モノレポ構成（`apps: api / bot / dashboard / worker`、`packages: db / types / shared / plugin-sdk / rule-engine / ui`）により、機能追加時の再利用性は高いが、横断品質（監視・運用）を次段階で強化する余地がある。  
- Plugin は 12 個が稼働し、コミュニティ運営で必要な機能の厚みは十分。一方で Stub 3 件（`hoyolink / voice_consent / external_server`）は有効化不可で、機能期待とのギャップが残る。  
- Rule Engine は Trigger 8 種 / Condition 9 種 / Action 8 種 + dry-run テスト API まで実装済みで、設定変更時の安全性は一定担保されている。  
- Analytics は生数値中心で、ダッシュボード上の可視化（グラフ UI）が未統合。価値の「発見性」が低く、意思決定に使いづらい状態。  
- Dashboard にリアルタイム更新（WebSocket/SSE）が無く、運用画面の即時性が不足。手動リロード依存は運用負荷と見逃しリスクを増やす。  
- API rate limit が in-memory 実装であり、本番マルチインスタンス運用では整合性が崩れる懸念があるため、Redis 化は高優先の運用品質課題。  
- Event バナー等のファイルアップロード未対応は、運営体験の完成度を下げる。管理画面の導線を閉じるためにも対応優先度は中以上。  

# 優先タスク（Must）

## 1. API Rate Limit の Redis 化（本番運用耐性）
- 背景（なぜ必要か）  
  マルチインスタンス環境で in-memory 制限はノードごとに分断され、過負荷防止が機能しない。障害・濫用リスクが高く、SLO 逸脱につながるため最優先。  
  - ユーザー価値: 安定稼働で応答遅延・エラーを減らす。  
  - 運用リスク低減: DDoS/スパイク時の防御を一貫化。  
- 実装対象  
  `apps/api`, `packages/shared`（共通ミドルウェア化する場合）, 必要に応じて `apps/worker`（同一制限戦略の適用確認）。  
- 完了条件（DoD）  
  1) Redis バックエンドで rate limit が動作。  
  2) 複数 API インスタンスで同一キーの制限が一貫。  
  3) 閾値超過時のレスポンス仕様をドキュメント化。  
  4) 負荷試験（要確認: 既存ツール）で期待挙動を確認。  
- 見積  
  M  
- 依存関係  
  Redis 接続設定、インフラ環境変数、運用側のレート閾値合意（要確認）。

## 2. Dashboard のリアルタイム更新導入（運用監視の即時性向上）
- 背景（なぜ必要か）  
  手動更新前提では通知・モデレーション・イベント状況の追従が遅れ、運営判断が後手になる。  
  - ユーザー価値: 状態変化の即時把握で運営体験を改善。  
  - 運用リスク低減: インシデント検知遅延を短縮。  
- 実装対象  
  `apps/dashboard`（UI 受信・状態更新）, `apps/api`（WebSocket/SSE エンドポイント）, 必要に応じて `apps/worker`/`apps/bot`（イベント発火）。  
- 完了条件（DoD）  
  1) 対象画面（要確認: まずは Analytics/Moderation）で自動更新。  
  2) 接続断時の再接続・フォールバック（手動更新）導線あり。  
  3) 更新イベントの最小スキーマ定義が `packages/types` に反映。  
  4) 監視ログで接続数・切断率を可視化。  
- 見積  
  L  
- 依存関係  
  配信方式の選定（WebSocket vs SSE）、認証方式、負荷見積、フロント状態管理方針。

## 3. 基本可観測性の強化（監視・アラート最小セット）
- 背景（なぜ必要か）  
  MVP 後の拡張期は不具合混入確率が上がる。メトリクス・構造化ログ・主要アラートがないと MTTR が長期化する。  
  - ユーザー価値: 障害復旧が早くサービス信頼性が向上。  
  - 運用リスク低減: 問題の早期検知・原因特定を可能にする。  
- 実装対象  
  `apps/api`, `apps/bot`, `apps/worker`, `apps/dashboard`（フロントエラー収集は要確認）, `packages/shared`（ロガー共通化）。  
- 完了条件（DoD）  
  1) 主要メトリクス（RPS, error rate, queue lag, job failure）を取得。  
  2) 構造化ログに trace/request ID を付与。  
  3) 最低 3 種のアラート閾値を設定し、検知テストを実施。  
  4) ランブック（一次対応手順）を docs 化。  
- 見積  
  M  
- 依存関係  
  監視基盤選定（要確認）、通知先（Slack/Discord）合意、SLO 仮決定。

# 優先タスク（Should）

## 1. Analytics グラフ UI 統合
- 背景（なぜ必要か）  
  生数値のみでは傾向把握が難しく、機能が「あるが使われない」状態になりやすい。  
  - ユーザー価値: トレンド可視化で施策判断を高速化。  
  - 運用リスク低減: 異常値の視認性向上で検知漏れを減らす。  
- 実装対象  
  `apps/dashboard`, `apps/api`（集計 API 拡張が必要なら）, `packages/ui`。  
- 完了条件（DoD）  
  1) 主要 KPI の時系列グラフを表示。  
  2) 期間切替（日/週/月）と空データ時 UI を実装。  
  3) 初期表示性能目標（要確認）を満たす。  
- 見積  
  M  
- 依存関係  
  KPI 定義の確定、既存集計データ粒度の確認。

## 2. Event バナー等ファイルアップロード対応
- 背景（なぜ必要か）  
  イベント告知品質が運営成果に直結する。画像アップロード未対応は利用体験の欠落。  
  - ユーザー価値: 告知の訴求力向上・管理工数削減。  
  - 運用リスク低減: 外部手作業の貼り替え運用を削減。  
- 実装対象  
  `apps/dashboard`, `apps/api`, `packages/types`, `packages/db`（メタデータ管理）, 保存先連携（要確認）。  
- 完了条件（DoD）  
  1) 許可形式/サイズ制限付きでアップロード可能。  
  2) 権限チェック（管理者のみ等）を実装。  
  3) 失敗時リトライ・エラーメッセージ整備。  
- 見積  
  M  
- 依存関係  
  オブジェクトストレージ選定（S3 等、要確認）、CDN 方針、ウイルススキャン要否。

## 3. Stub Plugin の実装優先度再評価と 1 件着手
- 背景（なぜ必要か）  
  Stub のまま公開されると期待不一致が増える。少なくとも 1 件を実装しロードマップの信頼性を高める。  
  - ユーザー価値: 新機能提供による導入動機増加。  
  - 運用リスク低減: 未実装機能問い合わせの削減。  
- 実装対象  
  `apps/bot`, `packages/plugin-sdk`, `packages/types`, 必要なら `apps/api`。  
- 完了条件（DoD）  
  1) Stub 1 件を有効化可能状態へ。  
  2) 設定 UI/API/実行処理が一貫して動作。  
  3) 最低限の利用ガイドを docs 追加。  
- 見積  
  M  
- 依存関係  
  どの plugin を先行するかの事業優先判断（要確認）。

# 2週間実行計画

## Week 1（設計・最小実装）
- マイルストーン 1: Rate limit Redis 化の設計確定と実装着手。  
  - API ミドルウェア差し替え、キー設計、閾値初期値決定（要確認）。
- マイルストーン 2: リアルタイム更新の方式決定（WebSocket/SSE）と PoC。  
  - 1 画面で push 更新を検証し、接続/再接続の基本動作を確認。
- マイルストーン 3: 可観測性の最小スキーマ決定。  
  - 共通ログ項目、必須メトリクス、アラートドラフトを定義。

- マイルストーン 4: Must タスクの統合試験。  
  - マルチインスタンス制限テスト、リアルタイム更新 E2E、アラート発火訓練。
- マイルストーン 5: Should タスクから 1 件を完了。  
  - 推奨: Analytics グラフ UI（短期で価値が見えやすい）。
- マイルストーン 6: 次スプリント準備。  
  - Stub plugin 先行対象の意思決定、ファイルアップロードの技術選定完了。

# チーム共有用サマリー（200〜300文字）

Lunaria は MVP 機能が揃い、運営 Bot とダッシュボードの基盤は実用段階です。次スプリントは「安定運用」と「意思決定速度」を主軸に、①API rate limit の Redis 化 ②ダッシュボードのリアルタイム更新 ③監視・アラート整備を Must として実施します。並行して Analytics のグラフ UI を優先着手し、運用負荷と見逃しリスクを下げつつ、ユーザーが価値を体感できる改善を進めます。
